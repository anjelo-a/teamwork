# Purpose

Force a final audit before claiming work is complete.

# When to use

Use when:

* implementation is done or nearly done
* code behavior changed
* API, DB, frontend architecture, or realtime logic changed
* preparing final summary or completion report

# When not to use

Do not use when:

* implementation has not started
* the task is docs-only
* the edit is trivial and non-behavioral

# Required behavior

* re-read the task
* re-check `AGENTS.md`, `PLANS.md`, and applicable skills
* inspect changed files
* verify task fidelity
* verify applicable safety, security, and performance rules
* verify tests and checks actually ran when applicable
* report risks honestly
* state partial verification honestly if needed

# Forbidden behavior

* saying “done” without review
* pretending tests passed if not run
* hiding uncertainty
* overstating readiness
* ignoring applicable repo rules

# Delivery checklist

* task fully matched
* relevant skills followed
* verification performed honestly
* remaining risks stated clearly
* completion summary accurate
